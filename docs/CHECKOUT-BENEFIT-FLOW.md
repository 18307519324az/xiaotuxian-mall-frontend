# 结算页权益流程说明

## 结算页模块

当前结算页不仅仅是“确认商品”页面，还承担：

- 地址确认
- 配送时间选择
- 支付方式选择
- 优惠券选择
- 礼品卡选择
- 金额联动计算
- 同步/异步下单请求发起

因此它是前端交易链路里最关键的页面之一。

## 优惠券选择流程

结算页会根据用户可用权益展示优惠券列表，用户选择后会立即影响：

- 商品优惠抵扣
- 最终应付金额
- 下单请求中的 `couponId`

这说明优惠券不是静态展示，而是直接参与提交请求的业务参数。

## 礼品卡使用流程

礼品卡在结算页的作用与优惠券类似，但表现形式不同。页面会展示：

- 已绑定礼品卡
- 可用余额
- 当前选择的礼品卡
- 礼品卡抵扣金额

提交请求时会带上 `giftCardCode`，后端再把权益快照落到订单中。

## 货到付款手续费

当前页面还支持货到付款模式，切换支付方式时会联动：

- 支付方式说明
- 货到付款手续费
- 最终应付金额

这也是公开展示时非常容易体现“页面不是静态模板”的部分。

## 金额计算公式

结算页金额摘要会聚合展示：

- 商品总额
- 运费
- 优惠券抵扣
- 礼品卡抵扣
- 货到付款手续费
- 最终应付金额

用户每次切换优惠券、礼品卡或支付方式时，这些字段都应该同步变化。

## 同步下单 payload

同步下单模式下，页面会在校验地址、商品和库存后发起普通下单请求。提交数据中重点字段包括：

- `addressId`
- `goods`
- `couponId`
- `giftCardCode`
- `payType`
- `payChannel`

这条路径更适合稳定演示和基础对账。

## 异步下单 payload

异步下单模式下，页面同样会带上权益参数，但后端返回的不是最终订单完成状态，而是处理中状态。前端需要处理：

- 请求发起成功
- 返回 `orderNo`
- 返回 `PROCESSING`

这使得页面逻辑比同步下单多了一层状态管理。

## 支付页展示

支付页会继续展示：

- 应付金额
- 优惠券抵扣
- 礼品卡抵扣
- 订单编号与待支付状态

这说明权益字段并没有停留在结算页，而是随订单继续传递。

## 订单详情展示

订单详情页会把结算阶段形成的权益快照继续展示出来，包括：

- 商品优惠
- 礼品卡抵扣
- 优惠券名称
- 礼品卡编号

这使读者可以直接验证“下单时的权益选择已经进入订单结果”。

## 常见问题

常见问题通常集中在：

- 优惠券与礼品卡是否能同时参与金额计算
- 货到付款手续费是否会重复叠加
- 异步下单时支付页是否能正确等待最终状态
- 订单详情页是否能回显结算时的权益字段

## 补充说明

异步下单后，前端还会继续轮询处理状态接口，直到出现以下结果之一：

- `SUCCESS`
- `FAILED`

如果成功，页面会继续进入支付页或订单详情；如果失败，页面要能提示用户重试或回退。

## 公开展示价值

这条链路最适合在面试或作品集里展示的点包括：

- 库存不是只在后端校验，前端也做闭环拦截
- 优惠券与礼品卡不是静态 UI，而是真实参与金额计算
- 货到付款手续费会联动影响支付金额
- 异步下单让页面需要承担轮询和结果展示

这些内容组合在一起，能显著提升前端仓库的项目完整度。
