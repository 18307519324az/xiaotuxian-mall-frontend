# 接口模式说明

## Mock 模式

Mock 模式下，前端请求会转发到本地 Mock Service。它更适合：

- 本地页面联调
- 快速演示完整流程
- 制作 README 截图
- 在缺少完整基础设施时运行页面

Mock 模式的优势是启动成本低、环境依赖少。

## 真实服务模式

真实服务模式下，前端请求对接主仓中的真实微服务接口。它更适合：

- 演示库存预扣
- 演示异步下单
- 演示订单状态轮询
- 演示真实订单与支付链路

这部分模式可以把页面层能力和后端 Redis、RabbitMQ、MySQL 链路连接起来。

## 代理配置

本地开发时通常通过前端开发代理转发 `/api/**` 请求。配置正确时：

- Mock 模式会转发到 `xiaotuxian-mall-mock-service`
- 真实服务模式会转发到主仓对应服务入口

因此切模式时首先要确认代理目标，而不是只改页面文案。

## 订单接口

前端在两种模式下都依赖订单相关接口，但关注点不同：

- Mock 模式：更强调字段兼容、页面稳定性和演示完整度
- 真实服务模式：更强调同步下单、异步下单和订单状态流转

公开展示时，订单接口是把购物车、结算、支付和订单详情串起来的核心节点。

## 库存接口

库存相关接口会影响：

- 分类页库存排序
- 商品详情页库存展示
- 购物车库存校验
- 结算页提交前复核

真实服务模式下，这部分能力还能进一步对应到 Redis Lua 预扣库存链路。

## 优惠券与礼品卡接口

权益接口主要用于支撑：

- 结算页优惠券列表
- 结算页礼品卡列表
- 下单请求中的 `couponId` 和 `giftCardCode`
- 支付页与订单详情页的权益回显

因此这些接口不是边缘功能，而是交易链路的一部分。

## 异步下单状态接口

异步下单场景下，前端除了提交订单接口外，还需要依赖状态查询接口来轮询：

- `PROCESSING`
- `SUCCESS`
- `FAILED`

这也是前端仓 README 中必须明确写出“异步下单轮询”的原因。

## 切换接口模式注意事项

如果只保留 Mock 模式，项目很难解释真实后端能力；如果只保留真实服务模式，页面演示成本会变高。双模式并存可以兼顾：

- 演示稳定性
- 技术说明深度
- 本地联调效率

## 结论

```text
浏览器页面
  -> /api/**
  -> 前端开发代理
  -> xiaotuxian-mall-mock-service
```

这种模式下，重点在于页面流程与字段兼容，而不强调真实中间件。

双模式不是重复建设，而是前端公开展示和真实后端讲解之间的平衡设计。这也是前端仓 README 中必须单独保留“接口模式”章节的原因。
